### SPLT-726 — Persistent login failure (NG user T231020201431puid79372630, v1.61.0)

**Context:** User still can't log in after updating to v1.61.0. Prior fix (SPLT-642) for 2FA/self-deactivation was expected to resolve it, didn't.

**Investigation approach:**
1. 直接查 `t_patron_user_device`，找到該 user 的 Android 裝置 status = `BLOCKED`。
2. 用 AI 翻 `sportybet-patron` 整個 codebase，確認只有一個 API 會產生 `BLOCKED` 結果：`/user/devices/blocking/execute`。
3. 以該 device id 為 key，搜尋 `update_time` 前後的 log，除了這筆 block 紀錄本身之外沒有其他相關 log。

**Conclusion (updated 2026-04-24 ~11:44):** Block 是 **user 自己透過 Device Management 功能封鎖自己的裝置**，不是 OP 或外部系統操作。

確認來源：Evans Lin（API 作者）說明 `/user/devices/blocking/execute` 這支 API 可以由 user 自己在 Device Management 介面呼叫。日誌裡沒有 OP 或內部排程的痕跡，與這個解釋吻合。

**Key learning:** 「外部呼叫」不一定是外部系統，也可能是 user 自己透過 feature 操作。調查 state 來源時，除了 operator/scheduler 路徑，也要確認有無 user-facing 管理功能能觸發同一條 code path。

**Debugging pattern used:**
- DB state → code path tracing（找哪條路徑能寫入這個 state）→ log cross-reference（確認有沒有內部觸發紀錄）→ 詢問 API 作者確認呼叫端
- 排除內部路徑的方式是 codebase scan，不是假設。

### SPLT-608 — NG Android "app only available in Nigeria" (user in Nigeria)

**Context:** User W230430023345puid47606020，NG region，在 Nigeria/Lagos 實體位置，無 VPN，仍收到 geo-restriction 錯誤。

**Error source:**
- Android 端觸發這個訊息的條件：呼叫 `/androidGooglePlayAvailable` 後回傳 `false`，或是收到 HTTP 451。
- patron codebase 內沒有 451 這個 error code 或相關訊息 → 推論 451 是 HTTP status，來自 gateway 或 upstream，不是 patron 業務邏輯。

**Backend logic:**
- `/androidGooglePlayAvailable` 的邏輯只讀 `X_COUNTRY_CODE` header，與該國家部署時設定的 config 比對。
- 不做任何 user-origin 的判斷，也不查使用者位置資料。
- 所以如果 header 不對，patron 沒有能力自我修正。

**Gateway hypothesis:**
- Gateway 可能根據 client IP 做 GeoIP lookup，並設定 / 覆寫 `X_COUNTRY_CODE`。
- 如果 gateway 的 GeoIP 判斷錯誤（IP 被標記為非 NG），patron 就會收到錯誤的 header 然後回傳 false。
- 這個假設的目的：釐清斷點到底在 client IP（GeoIP 錯誤）還是在其他層。

**Next step:** 確認 gateway 是否有讀 IP 做 GeoIP 並設定 country header 的邏輯。

### SPLT-679 — Archive rule review for 4 tables in afbet_patron

**Context:** 對 `afbet_patron` 下四張 table 做封存規則的 review，判斷是否能封存、封存條件為何。

**t_patron_cipher**
- **Verdict:** 暫不建議封存。
- Key reason: `update_time` 不是可靠的 inactivity signal；null `user_id` 不代表 cipher inactive；封存後可能 break 舊的 `ursId` lookup（`/ms/cipher`）；create path 無法完整 rebuild `symmetric_cipher`。
- 建議：等 `ursId` TTL/revoke 行為明確，或加上 explicit expiry/status field 再評估。

**t_patron_user_operation_log**
- **Verdict:** 保持現有規則（`create_time < DATE_SUB(CURDATE(), INTERVAL 179 DAY)`）。
- Key reason: write-once table，`idx_create_time` 已存在，封存操作上直觀；生產讀取多為最新資料或短時間窗口。
- Caveats: 兩條 unbounded device-lookup 路徑（suspicious-device scheduler、`/ms/users/devices`）封存後可能 miss 超過 179 天沒動的裝置 → 需要 product/risk sign-off；「最後登入時間」endpoint 實際上是 `MAX(create_time)` across all operation rows，封存後休眠 user 會回傳 null。
- 建議：規則本身不需改，重點是取得 unbounded device query 的 sign-off，並釐清 BO endpoint 的 login vs operation 語意。

**t_base_trans_cipher**
- **Verdict:** 可封存，blocked on 一個 open question。
- Key reason: `is_del` 在 code 裡從未被設成 1，不能作為封存 predicate；`create_time` 不是可靠 activity signal；只有 `update_time` 可用（最小 retention > 30 天，超過 Redis TTL）；`/ms/bizCipher` PK lookup miss 時回 200 `{ data: null }`，不 error 也不 auto-insert。
- Pending: 確認 `/ms/bizCipher` caller 在收到 null 時是否會觸發 re-registration（POST `/base/cipher`）。若是，封存安全；若否，retention N 必須覆蓋最長的 client 休眠週期。

**t_patron_user_trusted_device**
- **Verdict:** 現有 code 下不建議封存，有 3 個 preconditions。
- Key reason: `countAllByUserIdNotAndDeviceId` 無時間 predicate，封存後舊 device id 可能被重新允許 register 到新 user（安全問題）；windowed reads 用 `create_time`，retention 必須 ≥ effective `keepRecordForXDays`（code default 30 天）；沒有 standalone `idx_create_time`，range purge 會 full scan。
- Note: `update_time` 不能用，DDL 沒有 `ON UPDATE CURRENT_TIMESTAMP`，也沒有 UPDATE statement，frozen at insert。
- Preconditions: (1) 確認 device ownership check 是否需要 time-bound；(2) 確認 runtime `keepRecordForXDays` cap；(3) DBA 加 `idx_create_time`。

### SPLT-525 — BR/MX Status Communication Site Localisation

**Context:** Status site 新增 BR/MX 多語系支援，後端負責上傳翻譯檔、前端拉取。

**Post-deploy findings:**
- 翻譯檔已正常上傳到 S3 ✓
- BR default 群組拿到翻譯，但內容是英文 ✗
- Root cause: OP 把翻譯放在 `pt-br` key 下，但系統查的是 default language 的檔案；其他國家同樣受影響——Default 國家、Default 語言的 key 狀態都是 **draft**，導致 API 取預設值的檔案回傳空物件。

**Dependency:** CMS 需要完成翻譯 audit（將 draft 狀態的 key publish）後，API 才能正確回傳翻譯內容。

**Remaining validation:**
- CMS audit 完成後，確認後端有正確上傳翻譯檔。
- 確認前端能正確 fetch 到翻譯。

**Update 2026-04-24 ~15:00 — MX 翻譯檔缺失問題**

MX 沒有獨立的 country code 部署，跟 CM、MZ 一起掛在 `int` 底下。CMS 上 MX 的翻譯放在 `mx → default`，但系統只會產出 `int/translation/default.json`，MX 那份根本不會出現在 S3 上，前端拿不到。

**解法：** 把 CMS 上 `mx → default` 的翻譯搬到 `int → es-mx`，改設定讓後端多產出 `int/translation/es-mx.json`，前端改讀這份。不動 code，純設定 + CMS 搬家。

**Trade-off：** 另一種做法是改後端 code 硬繞，但等於把部署架構問題寫死在業務邏輯裡，MX 未來獨立部署又要再改。不建議。

**Action items：**
- [ ] CMS 上把 `mx → default` 翻譯複製到 `int → es-mx`
- [ ] 改 INT 環境 config 加入 `es-mx` 語言（⚠️ 改 `patron-internal`，不是 `patron`）
- [ ] 請 OP 重新 audit `int → es-mx` 的翻譯（早上才 audit 過，搬家後需要再一次）
- [ ] 前端改為讀取 `int/translation/es-mx.json`

**Pitfall (2026-04-24 ~18:20)：**
翻譯上傳是 scheduler 跑的，跑在 `patron-internal`（worker），不是 `patron`（app/API）。改了半天 config 都改在 `patron` 上，scheduler 根本沒讀到，所以什麼反應都沒有。

> **Key learning：** `patron` = API app，`patron-internal` = worker/scheduler。涉及排程任務的 config 要改 `patron-internal`。
